# 产品业务契约 v0.3

> 状态：已确认 | 最终版个人学习整理方向

---

## 1. 产品定位

本项目定位为个人使用的学习材料整理工具。

核心目标：

- 收集个人学习材料
- 整理成知识卡片
- 通过状态、标签、搜索找回内容
- 支持归档、删除恢复和基础统计

本项目不做公共内容分发，不做面向他人的信息流。

---

## 2. 审核避雷约定

### 2.1 禁用表达

页面标题、按钮、导航、占位文案中避免使用以下表达：

| 避免使用 | 推荐替代表达 |
|----------|--------------|
| AI日报 | AI知识卡片 / 技术笔记 |

| 个人中心 | 我的工作台 / 学习记录 |

### 2.2 产品边界

最终版仍不出现以下功能或文案：

- 新闻

- 资讯
- 日报
- 头条

- 推荐流

## 3. 最终版业务闭环

最终版主流程：

```text
粘贴学习材料
-> 智能整理
-> 生成标题 / 摘要 / 建议标签
-> 用户确认或编辑
-> 保存为知识卡片
-> 通过搜索、状态、标签找回
-> 回顾、归档、删除恢复
```

核心原则：

- 智能能力只服务于个人学习整理
- 不直接保存 AI 整理结果，必须由用户确认
- 不做公共内容抓取、公共推荐或外部信息流
- 标签体系以预设标签为主，避免复杂标签管理

## 5. 核心数据模型

### 5.1 KnowledgeCard

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string | 是 | 服务端生成 |
| userId | string | 是 | 服务端保存，用于关联当前登录用户；前端不传 |
| title | string | 是 | 卡片标题，最长 200 字符 |
| sourceText | string | 否 | 原文或摘录，默认空字符串 |
| summary | string | 否 | 摘要，默认空字符串 |
| sourceUrl | string | 否 | 来源链接，默认空字符串 |
| tags | string[] | 否 | 标签列表，默认空数组 |
| status | string | 是 | `todo` / `done` / `archived` |
| createdAt | string | 是 | ISO 8601 时间 |
| updatedAt | string | 是 | ISO 8601 时间 |
| deletedAt | string? | 否 | ISO 8601 时间；未删除时为 `null` |

### 5.2 状态含义

| status | 页面文案 | 说明 |
|--------|----------|------|
| todo | 待整理 | 已收集但还没整理 |
| done | 已整理 | 已整理为可复习卡片 |
| archived | 已归档 | 暂时不在日常整理视图中展示 |

删除不作为 `status` 的一种值，使用 `deletedAt` 表示软删除。

原因：

- 删除后恢复时，可以保留原来的 `status`
- 状态筛选逻辑更清晰
- 归档和删除语义不混在一起

### 5.3 用户隔离

所有卡片都必须归属于一个用户。

后端规则：

- `KnowledgeCard.UserId` 关联 `Users.Id`
- 创建卡片时，`UserId` 从当前 token 中解析，不由前端传入
- 查询列表、详情、更新、归档、删除、恢复时，只能操作当前登录用户自己的卡片
- 不允许通过接口访问其他用户的卡片

前端规则：

- 前端请求中不传 `userId`
- 前端只展示当前登录用户自己的数据

### 5.4 tags 传输与存储

接口请求和响应中，`tags` 统一使用字符串数组。最终版以预设标签为主，前端从预设列表中选择，后端仍按 `string[]` 保存：

```json
["AI", "学习", "工具"]
```

后端存储可以继续使用一列 JSON 字符串，例如 `TagsJson`：

```json
"[\"AI\",\"学习\",\"工具\"]"
```

后端负责在接口层完成转换：

- 接收请求时：`string[]` -> JSON 字符串
- 返回响应时：JSON 字符串 -> `string[]`
- 空标签返回 `[]`，不返回 `null`

### 5.5 时间格式

接口中的时间字段统一使用 ISO 8601：

```json
"2026-06-16T15:30:00+08:00"
```

后端返回：

- `createdAt`
- `updatedAt`
- `deletedAt`

前端只负责格式化展示，不改变接口原始含义。

## 6. 最终版核心能力

### 6.1 智能整理

用途：把用户粘贴的学习材料整理成可编辑的卡片草稿。

输入内容：

- 原文 / 摘录
- 可选来源链接

输出建议：

- 建议标题
- 建议摘要
- 建议标签
- 建议状态

前端交互：

- 用户点击“智能整理”或“提取摘要”
- 后端返回整理建议
- 前端填充到新建 / 编辑表单
- 用户确认或修改后，再调用创建 / 更新接口保存

后端规则：

- 智能整理接口只返回建议，不直接保存卡片
- 不抓取公共内容，不生成公共信息流
- 标签建议应优先从预设标签中选择
- 模型失败时返回明确错误，前端保留手动编辑能力

推荐文案：

- 智能整理
- 提取摘要
- 建议标签
- 生成标题

### 6.5 标签筛选

用途：按个人整理标签查看卡片。

前端交互：

- 内容池支持选择一个标签进行筛选
- 工作台展示常用标签入口
- 新建 / 编辑页仍使用文本输入，前端拆分为 `string[]`

后端规则：

- 标签筛选参数建议命名为 `tag`
- 最终版暂不要求单独建 Tags 表
- 常用标签可以从当前用户未删除卡片的 `tags` 聚合得出

---

### 6.6 软删除与回收站

删除语义：

- 普通删除是软删除
- 软删除设置 `deletedAt`
- 软删除后不出现在内容池、待整理、工作台统计中
- 软删除后可以在回收站中恢复

恢复语义：

- 恢复时清空 `deletedAt`
- 恢复后保留删除前的 `status`
- 恢复后刷新 `updatedAt`

彻底删除语义：

- 只有回收站中提供彻底删除
- 彻底删除前必须二次确认
- 彻底删除后数据库移除，不提供恢复

---

### 6.7 工作台统计增强

用途：让工作台从入口页变成个人整理概览。

展示数据：

- 待整理数量
- 最近整理数量
- 已归档数量
- 最近编辑卡片
- 常用标签

统计范围：

- 默认只统计当前用户数据
- 默认排除已删除卡片
- 最近整理可以按 `updatedAt DESC` 取前若干条

---

### 6.8 整理流程优化

待整理卡片操作：

- 卡片列表提供“去整理”
- 点击后进入编辑页
- 用户填写摘要、标签、状态
- 保存时可设置为 `done`

编辑页规则：

- 支持修改所有可编辑字段
- 状态可在 `todo` / `done` / `archived` 之间切换
- 不允许在普通编辑页直接设置删除状态

## 7. 建议接口变更

> 本节用于前后端对齐，最终以 API 接口文档为准。

### 7.1 卡片列表

```
GET /api/v1/cards?page=1&pageSize=20&status=all&keyword=&tag=&sort=updatedAt&order=desc
```

参数：

| 参数 | 类型 | 默认 | 说明 |
|------|------|------|------|
| page | int | 1 | 页码 |
| pageSize | int | 20 | 每页数量 |
| status | string | all | `all` / `todo` / `done` / `archived` |
| keyword | string | 空 | 搜索标题、摘要、原文、标签 |
| tag | string | 空 | 按单个标签筛选 |
| sort | string | updatedAt | `createdAt` / `updatedAt` |
| order | string | desc | `asc` / `desc` |

默认行为：

- 排除 `deletedAt != null` 的卡片
- 按 `updatedAt DESC` 排序

### 7.2 回收站列表

```
GET /api/v1/cards/deleted?page=1&pageSize=20&keyword=&sort=deletedAt&order=desc
```

默认行为：

- 只返回 `deletedAt != null` 的卡片
- 只返回当前用户数据

### 7.3 软删除卡片

```
DELETE /api/v1/cards/{id}
```

行为：

- 设置 `deletedAt`
- 刷新 `updatedAt`
- 不物理删除

### 7.4 恢复卡片

```
PATCH /api/v1/cards/{id}/restore
```

行为：

- 清空 `deletedAt`
- 保留原 `status`
- 刷新 `updatedAt`

### 7.5 彻底删除卡片

```
DELETE /api/v1/cards/{id}/permanent
```

行为：

- 仅允许删除当前用户自己的已软删除卡片
- 物理删除数据库记录

### 7.6 标签汇总

```
GET /api/v1/cards/tags
```

响应建议：

```json
{
  "code": 200,
  "data": [
    { "name": "AI", "count": 8 },
    { "name": "工具", "count": 5 }
  ]
}
```

默认行为：

- 只统计当前用户数据
- 排除已删除卡片
- 可以按 `count DESC` 排序

### 7.7 工作台概览

```
GET /api/v1/cards/overview
```

响应建议：

```json
{
  "code": 200,
  "data": {
    "todoCount": 3,
    "doneCount": 12,
    "archivedCount": 4,
    "deletedCount": 1,
    "recentCards": []
  }
}
```

说明：

- `recentCards` 使用普通卡片列表项结构
- 默认排除已删除卡片

---

### 7.8 智能整理

```
POST /api/v1/cards/organize
```

请求：

```json
{
  "sourceText": "用户粘贴的学习材料",
  "sourceUrl": "https://example.com/article"
}
```

响应建议：

```json
{
  "code": 200,
  "data": {
    "title": "建议标题",
    "summary": "建议摘要",
    "tags": ["RAG知识", "模型基础"],
    "status": "todo"
  }
}
```

行为：

- 只返回草稿建议，不创建卡片
- 前端确认后继续调用 `POST /cards`
- 标签建议优先从预设标签中选择

### 7.9 预设标签

```
GET /api/v1/tags/presets
```

响应建议：

```json
{
  "code": 200,
  "data": [
    "AI概念",
    "提示词工程",
    "RAG知识",
    "模型基础"
  ]
}
```

说明：

- 如果后端暂不提供该接口，前端可以先使用本地常量
- 不影响卡片保存结构，卡片仍提交 `tags: string[]`

### 7.10 搜索联想

```
GET /api/v1/cards/suggestions?keyword=rag
```

响应建议：

```json
{
  "code": 200,
  "data": {
    "keywords": ["RAG", "检索增强"],
    "tags": ["RAG知识"],
    "cards": [
      {
        "id": "card_a1b2c3",
        "title": "RAG 检索增强生成"
      }
    ]
  }
}
```

说明：

- 搜索范围限当前用户自己的未删除卡片
- 第一版可使用数据库模糊匹配，不要求 AI

## 8. 最终版不做

以下能力不进入最终版：

- 多用户公开浏览
- 内容推荐流
- 评论、点赞、转发
- 公开发布
- 排行榜
- 自动抓取外部内容
- 搜索外部内容
- 订阅公共来源
- 标签管理页
- 标签别名 / 标签合并
- 批量操作
- 分享卡片

### 8.1 后续版本暂缓项

以下需求有价值，但不进入最终版实现：

- 批量整理：后续可支持批量归档、批量删除、批量改状态
- 标签体系：后续可增加标签管理、标签合并、标签统计趋势
- 导出能力：后续可支持导出为 Markdown 或文本

## 10. 已确认事项

以下问题已由后端确认：

1. `KnowledgeCard` 增加 `DateTime? DeletedAt` 字段，`null` 表示未删除，非 `null` 表示已软删除。
2. `DELETE /cards/{id}` 从物理删除改为软删除。
3. 彻底删除接口使用 `DELETE /cards/{id}/permanent`。
4. 标签汇总由后端聚合，接口为 `GET /cards/tags`。
5. 工作台概览使用独立接口 `GET /cards/overview`。
